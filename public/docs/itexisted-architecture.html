Protocol as Interface — End-to-End Architecture
Version 1.0  ·  28 February 2026  ·  Umarise

Premise

"The app does not explain itself. The action is the interface.
Two screens, one loop. No menu, no navigation, no features — only a repeatable act."

itexisted.app is a stateless, link-first anchoring utility. It proves that a file existed at a specific moment in time by creating an immutable cryptographic record anchored to the Bitcoin blockchain via OpenTimestamps. The interface embodies the protocol: no accounts, no gallery, no stored data — only a repeatable act and its persistent proof.

Data Flow

What crosses the network boundary

Data Direction Destination
SHA-256 hash Client → API Umarise Core (origin_attestations table)
Passkey public key Client → localStorage Device-local only (credential ID + public key)
Passkey signature Client → certificate.json Included in ZIP bundle (Layer 2 proof)
OTS proof API → Client Bitcoin timestamp proof (.ots file)

What never crosses the network boundary

Critical Invariant
  • Original file bytes — hashed in RAM, never transmitted
  • File name — not sent, not stored
  • File metadata — not sent, not stored
  • User identity — no accounts, no email, no phone

Proof Layers

Layer Mechanism Proves Status
Layer 0 SHA-256 hash File content integrity (exact bytes) Immediate
Layer 1 OpenTimestamps → Bitcoin Temporal existence (before block N) ~4–24h
Layer 2 WebAuthn passkey signature Device-bound authorship Immediate
Layer 3 Certified attestation Third-party legal witness Optional (€4.95)

Evidence Bundle (ZIP)

Contents
  • artifact.* — the original file (re-uploaded by user before download)
  • certificate.json — origin_id, hash, timestamp, passkey signature, proof status
  • VERIFY.txt — human-readable verification instructions
  • proof.ots — OpenTimestamps binary proof (when anchored)

The ZIP is self-contained: any third party can independently verify the proof without contacting Umarise. The VERIFY.txt includes step-by-step instructions for manual verification using standard command-line tools.

Constitutional Constraints

Interface Test
  • The app never explains itself — discovery is external (spec, docs, context)
  • No onboarding, no tutorial, no "what is this"
  • Every addition must survive: "Is this the act, or explanation of the act?"
  • Verification is not a feature — it is inherent to the proof
  • Single upload: file passes via RAM to proof page — re-upload only as fallback for returning users
  • The proof URL (/proof/{TOKEN}) is the product — shareable, persistent, link-first
Core Boundary (C0)
  • Core accepts hash-only input — no bytes cross into infrastructure
  • Core remains valid if all companion apps (incl. itexisted.app) disappear
  • The companion provides convenience — the Core provides truth
  • No mechanism or database column exists to store file content

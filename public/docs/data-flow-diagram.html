Overview

Two paths produce the same result. Content never leaves the device. Only SHA-256 hashes cross any network boundary.

Full Data Flow

Path A — B2B (Partner API)

Partner system: SHA-256 hash van bestand berekenen (lokaal) ↓ POST /v1-core-origins + X-API-Key → Cloudflare WAF + TLS ↓ Edge Function: API key valideren → partner_api_keys ↓ Edge Function: rate limit checken → core_rate_limits ↓ INSERT hash + origin_id + timestamp → origin_attestations (write-once) ↓ OTS Worker leest unstamped hashes → Hetzner (cron, elk uur) ↓ Worker: Merkle-batch submitten → OTS Calendar Servers (3×) ↓ Calendar: anchor in Bitcoin → Bitcoin Blockchain ↓ Worker: .ots proof opslaan → core_ots_proofs (immutable)

Path B — B2C (itexisted.app)

itexisted.app: SHA-256 hash van bestand berekenen (in-browser) ↓ itexisted.app: INSERT page met hash → pages (Supabase RLS, device_user_id) ↓ DB trigger: bridge_page_to_core → automatisch ↓ INSERT hash + origin_id + timestamp → origin_attestations (write-once) ↓ … daarna identiek aan Path A (OTS Worker → Bitcoin) ↓ itexisted.app: proof ophalen → v1-core-resolve (origin_id lookup) ↓ itexisted.app: ZIP genereren (client-side) → certificate.json + artifact + .ots proof ↓ Gebruiker: ZIP downloaden → volledig lokaal, zero-storage

Verificatie (drie tracks, publiek)

Track 1 — In-app (itexisted.app) ZIP of bestand droppen → client-side hash check + registry lookup ↓ POST /v1-core-verify → lookup in origin_attestations ↓ Result: origin_id, captured_at, proof_status → weergave in UI Track 2 — Referentie Verifier (verify-anchoring.org) ZIP uploaden → 100% client-side, zero-API ↓ V(B, P, L) verificatie → SHA-256 + .ots parsing + Bitcoin block check ↓ Result: onafhankelijk bewijs → geen Umarise dependency Track 3 — CLI Scripts verify-anchor.sh / verify-anchor.py → zero-dependency, offline ↓ ots verify + sha256sum → terminal output

Public API Endpoints (geen API key)

POST /v1-core-verify → hash lookup in origin_attestations GET /v1-core-resolve → origin_id lookup (first-in-time) GET /v1-core-proof → download .ots uit core_ots_proofs GET /v1-core-health → DB ping + uptime status

Monitoring

OTS Worker → append naar attestation-ledger (JSONL) sync-attestations.js → vergelijk DB vs ledger (cron, elke 5 min) PostgreSQL → DDL events → core_ddl_audit

Critical Invariant

No column for content exists in any table. This is architectural, not policy. There is no mechanism to store file content even if someone tried. The hash is a one-way function: content cannot be derived from it.

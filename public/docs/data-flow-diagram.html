Data Flow Diagram

Umarise Core API — Two paths, one truth. Content never leaves the device.
Version 1.0 17 February 2026 Status: For External Reviewers

Overview

Two paths produce the same result. Content never leaves the device. Only SHA-256 hashes cross any network boundary.

Full Data Flow

Path A — B2B (Partner API)

Client: SHA-256 hash van bestand berekenen (lokaal) ↓ POST /v1-core-origins + X-API-Key → Cloudflare WAF + TLS ↓ Edge Function: API key valideren → partner_api_keys ↓ Edge Function: rate limit checken → core_rate_limits ↓ INSERT hash + origin_id + timestamp → origin_attestations (write-once) ↓ OTS Worker leest unstamped hashes → Hetzner (cron, elk uur) ↓ Worker: Merkle-batch submitten → OTS Calendar Servers (3×) ↓ Calendar: anchor in Bitcoin → Bitcoin Blockchain ↓ Worker: .ots proof opslaan → core_ots_proofs (immutable)

Path B — B2C (App via Supabase RLS)

Client: SHA-256 hash van bestand berekenen (lokaal) ↓ App: INSERT page met hash → pages (Supabase RLS) ↓ DB trigger: bridge_page_to_core → automatisch ↓ INSERT hash + origin_id + timestamp → origin_attestations (write-once) ↓ … daarna identiek aan Path A (OTS Worker → Bitcoin)

Verificatie (publiek, geen API key)

POST /v1-core-verify → lookup in origin_attestations GET /v1-core-resolve → lookup (first-in-time) in origin_attestations GET /v1-core-proof → download .ots uit core_ots_proofs GET /v1-core-health → DB ping

Monitoring

OTS Worker → append naar attestation-ledger (JSONL) sync-attestations.js → vergelijk DB vs ledger (cron, elke 5 min) PostgreSQL → DDL events → core_ddl_audit

Critical Invariant

No column for content exists in any table. This is architectural, not policy. There is no mechanism to store file content even if someone tried. The hash is a one-way function: content cannot be derived from it.

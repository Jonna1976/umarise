Anchoring Specification (IEC)

Version 1.0
Status Normative
Date 2026-02-26
URI https://anchoring-spec.org/v1.0/

1 Scope

This specification defines anchoring as a cryptographic method for establishing existence-at-or-before-T of a byte sequence.

It defines:

  • the verification function
  • the permitted outputs
  • the semantic scope of anchoring
  • the exclusion of non-chronological claims
  • ledger qualification requirements

This specification is normative.

2 Definition

Anchoring establishes one property only:
A specific byte sequence existed on or before time T.

No additional claims are implied.

Time T is derived from publicly verifiable ledger consensus. It is not self-asserted.

3 Verification Function

V(B, P, L) → { valid | invalid | unverifiable }
B
The exact byte sequence of the artifact under verification.
P
The proof bundle: a certificate containing a cryptographic hash commitment, and optionally a ledger proof.
L
Publicly verifiable ledger infrastructure (e.g. Bitcoin blockchain via OpenTimestamps).

Output semantics are defined in Section 4.

4 Output Semantics

OutputCondition
valid Hash integrity confirmed and ledger inclusion verified. The byte sequence existed on or before time T.
invalid Hash mismatch between artifact and certificate, or ledger proof is demonstrably incorrect.
unverifiable Verification cannot be completed. Possible causes: missing artifact, missing or incomplete ledger proof, inaccessible ledger infrastructure.

No other outputs are permitted.

An implementation must not return intermediate, partial, or qualified results outside this set.

5 Exclusions

Anchoring does not establish:

  • authorship
  • ownership
  • originality
  • legal enforceability
  • identity
  • intent
  • truthfulness

An implementation must not present anchoring results in a manner that implies any of the above.

6 Independence Requirement

A compliant anchoring proof must remain verifiable without reliance on the issuing entity.

Verification must not require:

  • account access
  • issuer-controlled infrastructure
  • issuer-controlled databases
If verification depends on the issuing service, the proof is non-compliant.

7 Ledger Qualification

A ledger qualifies for anchoring verification if:

  • it is publicly accessible without credentials
  • its state is determined by distributed consensus
  • its history is append-only and tamper-evident
  • timestamp derivation is based on block inclusion, not self-assertion

The Bitcoin blockchain, accessed via the OpenTimestamps protocol, satisfies these requirements.

8 Proof Bundle Structure

A conformant proof bundle (P) contains at minimum:

  • a certificate with a SHA-256 hash commitment
  • the original artifact (byte sequence B)

Optionally, a proof bundle may include:

  • an OpenTimestamps proof file (.ots) for ledger verification
  • an attestation certificate from a qualified attestant
  • a human-readable verification guide (VERIFY.txt)
Conformance note. An implementation that extends the outputs defined in Section 4, omits the exclusions defined in Section 5, or requires issuer infrastructure for verification as prohibited by Section 6, is non-conformant with this specification.
